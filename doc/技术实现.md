# 技术实现（维护者参考）

本文依据当前的 `dotfilesmanager` 包描述行为。`dfm` 是一个以**移动原文件到集中目录，再在原安装位置创建符号链接**为工作模型的 dotfile 管理器；并非复制同步工具。

## 打包、依赖与入口

项目使用 `pyproject.toml` 打包，控制台入口为 `dotfilesmanager.cli:main`；顶层 `dfm.py` 仅保留为兼容入口。运行时依赖如下：

|依赖|用途|
|---|---|
|`docopt`|从 CLI 用法字符串解析命令行|
|`pyyaml`|以 `SafeLoader` 读取、以 `SafeDumper` 写入 YAML 配置|
|`inquirer`|TTY 中的平台与路径目标向导|
|`platformdirs`|跨平台配置/数据路径候选生成|

`dfm setup` 是 Windows 专用的独立命令，不解析仓库或配置。其他变更命令加载配置、完成预检后直接执行操作并按既有规则保存配置。

## CLI 契约

```text
dfm add <install_path> [--system] [--non-interactive] [--target=SYSTEM=PATH]... [--dry-run] [--force]
dfm rm <path> [--all] [--dry-run] [--force]
dfm install [<save_path>] [--dry-run] [--force]
dfm share <save_path> <install_path> [--non-interactive] [--target=SYSTEM=PATH]... [--dry-run] [--force]
dfm view [--dry-run] [--force]
dfm doctor
dfm setup
```

|命令|参数含义|主要结果|
|---|---|---|
|`add`|现有的安装路径；`--system` 表示保存副本与当前系统相关|移动对象入仓库，并在原位置建链接|
|`rm`|仓库保存路径、指向仓库的符号链接路径，或 `view/` 下直接指向哈希保存命名空间的链接；可加 `--all`|TTY 选择要删除的已登记平台，默认当前系统；`--all` 删除全部登记和保存对象|
|`install`|可选的仓库保存路径或 `view/` 下直接指向哈希保存命名空间的链接|按当前系统登记重建一个或全部链接|
|`share`|已有仓库保存路径（或 `view/` 下直接指向哈希保存命名空间的链接）与新的安装路径|把该保存对象登记给当前系统并建链接|
|`view`|无，可加 `--force`|直接重建当前系统的可读链接视图；失败可能留下部分视图|
|`doctor`|无|只读诊断配置、保存对象和当前系统安装链接|

路径进入各命令前会做 `expanduser`、`normpath` 和 `abspath`。变更命令先校验 `dfm.yaml` 的顶层映射、相对保存路径、系统项及记录的 `path` 类型；所有平台记录只做这些结构校验，只有**当前平台**的安装路径才按本机 home/根目录规则校验，避免把其他系统路径当作本机错误。`rm`、指定保存项的 `install` 与 `share` 可将 `view/` 下的直接符号链接解析为规范保存路径：立即链接目标（相对目标按链接父目录解析）必须位于根目录内，首段为 32 位小写十六进制哈希，且为现有文件或目录；不会以 `realpath` 替换该规范目标。目录链接的后代、悬挂链接及指向根目录外、配置/元数据或非哈希路径的链接不会解析，仍被拒绝。`add` 还要求目标是文件或目录、位于 home、未在根目录内且计算出的保存路径尚不存在。`rm` 只接受位于根目录中的路径，或解析后指向该根目录的链接，并拒绝覆盖普通文件、目录或指向其他保存对象的链接。指定的 `install`/`share` 保存路径必须位于根目录且为现有文件或目录；`share` 的安装目标必须位于 home 且不在根目录内。

`add` 和 `share` 在 TTY 默认运行 `inquirer` 目标向导；非 TTY 必须给出 `--non-interactive`。`rm` 在 TTY 使用复选框列出该保存项的所有已登记系统（包括未知平台键），默认只选当前系统；取消不产生变更，空选择为 no-op。`--all` 跳过该选择并移除全部登记；非 TTY 的普通 `rm` 保持仅移除当前系统。自动化调用使用重复的 `--target=SYSTEM=PATH`，平台集合和顺序固定为 `linux`、`darwin`、`windows`、`android`。目标按第一个 `=` 切分并按目标平台作安全路径校验。向导、目标合并和覆盖确认全部在直接变更前完成；取消、EOF、缺少回答或拒绝确认均不会产生变更，dry-run 仅跳过最终确认。

候选转换先使用当前平台对应的 `platformdirs` provider 分类**当前**来源：Linux/Android
使用 Unix，Darwin 使用 MacOS，Windows 的 CONFIG 使用 Roaming、其余使用 Local。公开
category root 只允许用于这一步，且会尊重当前 XDG 覆盖；只保留 config/data 的同根别名，
顺序为 config、data。随后目标转换仍只使用固定字面模板及私有 append helper：Windows
CONFIG 为 Roaming，DATA 为 Local；macOS 会先为每个识别类别提供对应 Unix 候选（CONFIG 为
`~/.config`、DATA 为 `~/.local/share`），再提供 Application Support 模板并生成去重并集，
因此 Unix 候选为默认值。其他平台在 CONFIG 模板之后额外加入 Unix `~/.config` 候选，最后才
加入直接来源回退；
LOG 和其他未分类来源只有直接回退；CACHE/STATE 只有在根目录可区分时才直接回退，
同 DATA 根的 macOS/Windows 别名仍按 DATA 处理。绝不将当前重定向/XDG/AppData 环境当作
外部机器路径，重定向目标应由 Custom path 明确输入。

路径 List 默认选中第一个生成的建议路径，末尾只有 Custom path；没有 Skip 选项。Custom path
留空或只输入空白会返回该系统的路径选择。

Android 检测优先使用 `sys.getandroidapilevel()` 返回的整数，其次要求同时存在
`ANDROID_ROOT` 和 `ANDROID_DATA`，最后识别 `platform.system() == "android"`；
`TERMUX_VERSION` 单独存在不参与检测。Android/Termux 使用唯一 `android` YAML
键，不会回退读取旧的 `linux` 映射，也不会创建 `termux` 命名空间。这是破坏性变更：
先备份并提交仓库；保留真实 Linux 共享使用的 `linux` 映射，并在 Android 上另加
`android` 映射。仅既有 Termux 专用 `linux` 映射和平台专用 `linux` 对象必须手动
重新登记或迁移。

目标等价比较也使用**目标**平台语义：Windows 忽略大小写并把历史记录中的 `\`
和新记录中的 `/` 视作同一分隔符，但保留 YAML 中已有的字符串，不为幂等比较重写
数据。Windows 来源路径在向导前会规范为 POSIX 的 `~/...`，因此
`C:\Users\Alice\.config\nvim` 也能识别 `.config` 候选。Windows 的
`~/DOTFILES/...` 同样被拒绝。

`--dry-run` 读取并校验配置，执行本命令可用的路径/保存对象预检后打印命令级预览；不创建根目录、不写入任何文件，也不询问覆盖。`--force` 将安装/共享覆盖确认视为接受；对 `rm` 则允许直接替换冲突安装目标。

## 存储与配置生命周期

### 根目录和保存路径

根目录固定为：

```python
dotfiles_root = os.path.join(os.path.expanduser("~"), "dotfiles")  # ~/dotfiles
```

对安装路径先缩写 home（通常成为 `~`），再以其**父目录字符串**计算 MD5；最后拼接 basename。路径形式为：

```text
~/dotfiles/<md5(缩写后安装路径的dirname)>[/<platform.system().lower()>]/<basename>
```

未指定 `--system` 时，中间系统名为空，`os.path.join` 不会形成额外目录；指定后才插入如 `linux`、`darwin` 或 `windows`。因此同名但父目录不同的对象通过 MD5 目录区分；系统相关对象还通过系统名区分。

配置中的保存路径始终是相对 `~/dotfiles` 的路径，并在写入前将本机分隔符转换为内部 **POSIX `/`** 分隔符。安装时再把 `/` 转回本机 `os.sep`。

### YAML 格式

唯一读取和写入的配置文件是 `~/dotfiles/dfm.yaml`。其逻辑结构为：

```yaml
dotfiles:
  <相对保存路径（使用 /）>:
    <当前系统小写名>:
      path: <安装路径，home 可写为 ~>
```

示例（MD5 值仅为示意）：

```yaml
dotfiles:
  a1b2c3d4/linux/nvim:
    linux:
      path: ~/.config/nvim
  e5f6g7h8/.gitconfig:
    linux:
      path: ~/.gitconfig
```

不存在 `dfm.yaml` 时，内存初值为 `{"dotfiles": {}}`；已有 YAML 缺少顶层 `dotfiles` 时会补空字典。保存会创建根目录，将 YAML 写入根目录中的临时文件、`fsync` 文件后用 `os.replace` 原子替换 `dfm.yaml`，再尽力 `fsync` 目录。空文件、非映射 YAML 或错误的嵌套结构仍没有容错处理。

## 命令逐步实现

### `add`

1. 规范化安装路径，执行上述合法性、home 范围和保存路径冲突检查。
2. 计算保存绝对路径，创建其父目录。
3. `shutil.move(install_path, abs_save_path)` 移走原文件或目录。
4. 用 `os.symlink(abs_save_path, install_path)` 在原位置创建绝对目标链接。
5. 生成 POSIX 相对保存路径，在 `dotfiles[rel][当前系统]['path']` 写入缩写后的安装路径。
6. `main()` 将配置写入 `dfm.yaml`。

在第 1 步之后，普通 `add` 会在 TTY 为每个未配置的外部平台运行目标向导；
`--system` 跳过该步骤。非交互调用不读 stdin，而是在直接变更前解析、校验和合并
所有 `--target`。向导的最终确认、安装覆盖确认和 install 批量覆盖确认也都发生在
在执行直接操作前；传给操作层的确认回调只会接受已经批准的计划。

### `install`

1. 有 `<save_path>` 时，先确认其位于根目录且是现有文件或目录，转为相对保存路径，并确认当前系统在配置中有登记；否则打印 `is not kept in dotfiles`。
2. 遍历所有配置条目（或仅指定项），跳过未登记当前系统的条目；在任何覆盖提示或文件修改前，确认每个当前系统保存对象均存在且为文件或目录。
3. 用保存根目录加内部 POSIX 相对路径还原真实保存路径。
4. 调用 `_make_link(saved, install)` 创建链接；成功才打印 `Install ... -> ...`。

`_make_link` 会创建安装位置的父目录。若链接位置已存在（包含悬挂链接），它询问 `文件 ... 已存在，是否替换？(y/N)`；仅输入大小写不敏感的 `y` 才继续。`install` 会先完成整批覆盖确认，再开始替换；文件或链接用 `os.remove` 删除，目录用 `shutil.rmtree` 递归删除，然后创建符号链接。

### `share`

1. 先确认给定保存路径位于根目录且为现有文件或目录，将其转换为相对 POSIX 路径；安装目标必须位于 home 且不在根目录内。若不在配置顶层，打印 `is not kept in dotfiles` 并返回。
2. 若安装路径需要覆盖，先在直接变更前询问；拒绝则直接返回。
3. 调用同一 `_make_link` 将安装位置链接到给定保存路径；成功后才在该条目当前系统下写入新的安装路径并打印 `share ... -> ...`。
4. 若用户拒绝覆盖，不写入当前系统登记，也不重写配置。

`share` 不要求保存对象由其他平台创建；“共享”语义由已有配置条目和当前系统登记决定。但保存对象必须真实存在且为文件或目录。

`share` 的本地状态矩阵为：当前登记缺失且链接正确时仅登记；登记匹配且链接正确时
no-op；登记匹配但链接缺失时重建；登记匹配但目标是普通对象或错误链接时，TTY 在
直接操作前询问，非交互必须使用 `--force`。已有不同当前平台路径始终失败。平台专用
保存对象跳过目标向导，拒绝显式外部目标，且只允许 owner 平台 share。`share` 与
`install` 在破坏性替换前立即重新核对链接状态；状态变化会使操作失败，不会重新确认。

### `rm`

1. 若传入路径本身是链接，用 `os.readlink` 取得其目标作为保存路径；否则直接将参数视为保存路径。
2. 计算相对 POSIX 保存路径。TTY 中以复选框选择要移除的已登记系统，默认当前系统；非 TTY 默认当前系统，`--all` 直接选择全部。
3. 未选择当前系统时，不访问任何安装路径；若选中平台后仍有登记，只删除其配置项；若清空最后一个登记，则删除保存对象。
4. 选择当前系统时，只删除指向该保存对象的受管链接；普通文件、目录或指向其他位置的链接会使操作失败。使用 `--force` 时直接删除冲突对象，并同时删除选中的其他平台登记。
5. 若仍有其他系统配置，则把保存对象复制到当前安装位置（文件 `copy`，目录 `copytree`）；否则把保存对象移动回安装位置，并删除整个保存路径配置项。`--all` 在当前系统已登记时始终把保存对象移动回该本地安装位置，然后删除整个配置项；当前系统未登记时直接删除保存对象，且不访问外部平台安装路径。
6. 写回配置；保存对象的空父目录保留。

## 跨平台、权限与风险

* 非 Windows 使用标准 `os.path.expanduser` 展开 `~`。Windows 使用自定义逻辑：只要路径以 `~` 开头，就将第一个 `~` 替换为 `os.path.expanduser('~')`；它不只匹配 `~/` 或 `~\\`。路径缩写则只是文本前缀匹配 home 后替换为 `~`。
* Windows 普通命令不要求管理员。每次用户可见的链接创建保留 `target_is_directory`，且仅将 Windows `ERROR_PRIVILEGE_NOT_HELD`（WinError 1314）分类为可通过 `dfm setup` 处理的失败；其他原生错误保持不变。`setup` 先在临时目录实际测试文件和目录链接。仅当该测试得到 1314 时，才在 64 位注册表视图设置 `HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\AppModelUnlock\AllowDevelopmentWithoutDevLicense` DWORD 为 `1`，不设置 `AllowAllTrustedApps`。非提升进程只通过 UAC 提升绝对系统目录的 `reg.exe` 子进程及固定参数，等待其退出、读取设置并再次探测；取消、子进程失败、读取或复测失败均不报告成功。
* 保存哈希使用缩写后父目录的 UTF-8 MD5；缩写行为、home 表示及路径大小写/规范化差异会影响结果。路径范围校验使用规范化后的路径边界比较；Windows 上会忽略大小写。
* 保存路径和链接目标大多是绝对路径，迁移 home 目录或仓库后不会自动修复。

## 直接操作与排障

`add`、`rm`、`share` 与 `install` 直接执行预检后的文件系统操作，再按既有条件原子写入 `dfm.yaml`。普通变更仍拒绝 `dfm.yaml`、dotfiles root、生成的 `view/` 命名空间和现有符号链接父目录。异常（包括 Windows 1314）可能留下部分完成状态：先运行 `dfm setup`（仅 1314），检查并修复状态，再重试。历史 `.dfm-transaction.yaml`、`.dfm-backups` 和 `.dfm.lock` 被忽略。

`view` 同样是直接重建，不写 YAML；`--force` 替换已有视图目录。视图创建失败可能留下部分输出。若 Windows 链接创建报 1314，先运行 `dfm setup`，检查并修复部分视图，再重试。

`doctor` 是只读诊断：报告配置结构或路径错误、缺失保存对象、未引用保存对象以及当前系统安装链接问题；不创建缺失 root、不修复链接，也不读取或清理历史文件。

排障建议：

1. 先执行 `dfm doctor` 了解配置和链接状态；历史文件会被忽略，需人工决定是否保留或删除。
2. 确认当前用户的 `~/dotfiles/dfm.yaml` 存在且 YAML 结构符合上述 `dotfiles → 保存路径 → 系统名 → path` 层级。
3. 使用当前 `platform.system().lower()` 对应的配置项；其他系统的登记会被 `install` 跳过。
4. 检查保存路径是否使用 `/`，以及对应的仓库对象和安装路径是否真实存在；相对链接目标会相对链接所在目录解析。
5. Windows 若链接创建报 1314，运行 `dfm setup`，检查并修复任何部分完成状态后再重试；其他 Windows 错误按原始错误排查。
6. 对覆盖提示默认回答 `N`，先人工检查目标类型和内容；直接操作失败后按配置谨慎手动修正。
