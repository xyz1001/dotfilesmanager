# 技术实现（维护者参考）

本文依据当前的 `dotfilesmanager` 包描述行为。`dfm` 是一个以**移动原文件到集中目录，再在原安装位置创建符号链接**为工作模型的 dotfile 管理器；并非复制同步工具。

## 打包、依赖与入口

项目使用 `pyproject.toml` 打包，控制台入口为 `dotfilesmanager.cli:main`；顶层 `dfm.py` 仅保留为兼容入口。运行时依赖如下：

|依赖|用途|
|---|---|
|`docopt`|从 CLI 用法字符串解析命令行|
|`pyyaml`|以 `SafeLoader` 读取、以 `SafeDumper` 写入 YAML 配置|

`main()` 在 Windows 先检查管理员权限。变更命令在持锁状态下先恢复遗留事务、加载配置、完成预检并执行操作；成功提交后才输出成功消息。

## CLI 契约

```text
dfm add <install_path> [--system] [--dry-run] [--force] [--backup]
dfm rm <path> [--dry-run] [--force] [--backup]
dfm install [<save_path>] [--dry-run] [--force] [--backup]
dfm share <save_path> <install_path> [--dry-run] [--force] [--backup]
dfm doctor [--repair]
```

|命令|参数含义|主要结果|
|---|---|---|
|`add`|现有的安装路径；`--system` 表示保存副本与当前系统相关|移动对象入仓库，并在原位置建链接|
|`rm`|仓库保存路径，或指向仓库的符号链接路径|删除当前系统的登记；恢复文件或保留共享副本|
|`install`|可选的仓库保存路径|按当前系统登记重建一个或全部链接|
|`share`|已有仓库保存路径与新的安装路径|把该保存对象登记给当前系统并建链接|
|`doctor`|可选 `--repair`|诊断事务、配置、保存对象和当前系统安装链接；仅恢复有效事务或安全重建明确错误的链接|

路径进入各命令前会做 `expanduser`、`normpath` 和 `abspath`。变更命令先校验 `dfm.yaml` 的顶层映射、相对保存路径、系统项及记录的 `path` 类型；所有平台记录只做这些结构校验，只有**当前平台**的安装路径才按本机 home/根目录规则校验，避免把其他系统路径当作本机错误。`add` 还要求目标是文件或目录、位于 home、未在根目录内且计算出的保存路径尚不存在。`rm` 只接受位于根目录中的路径，或解析后指向该根目录的链接，并拒绝覆盖普通文件、目录或指向其他保存对象的链接。指定的 `install`/`share` 保存路径必须位于根目录且为现有文件或目录；`share` 的安装目标必须位于 home 且不在根目录内。

`--dry-run` 读取并校验配置，执行本命令可用的路径/保存对象预检后打印命令级预览；不获取锁、不创建根目录、不写入任何文件，也不询问覆盖。`--force` 将安装/共享覆盖确认视为接受；对 `rm` 则允许在事务快照保护下替换冲突安装目标。它不绕过保存对象、路径、日志或恢复校验。`--backup` 使成功事务保留其开始前快照及清单。

## 存储与配置生命周期

### 根目录和保存路径

根目录固定为：

```python
dotfiles_root = os.path.join(os.path.expanduser("~"), "dotfiles")  # ~/dotfiles
```

对安装路径先缩写 home（通常成为 `~`），再以其**父目录字符串**计算 MD5；最后拼接 basename。路径形式为：

```text
~/dotfiles/<md5(缩写后安装路径的dirname)>[/<platform.system().lower()>]/<basename>
```

未指定 `--system` 时，中间系统名为空，`os.path.join` 不会形成额外目录；指定后才插入如 `linux`、`darwin` 或 `windows`。因此同名但父目录不同的对象通过 MD5 目录区分；系统相关对象还通过系统名区分。

配置中的保存路径始终是相对 `~/dotfiles` 的路径，并在写入前将本机分隔符转换为内部 **POSIX `/`** 分隔符。安装时再把 `/` 转回本机 `os.sep`。

### YAML 格式

唯一读取和写入的配置文件是 `~/dotfiles/dfm.yaml`。其逻辑结构为：

```yaml
dotfiles:
  <相对保存路径（使用 /）>:
    <当前系统小写名>:
      path: <安装路径，home 可写为 ~>
```

示例（MD5 值仅为示意）：

```yaml
dotfiles:
  a1b2c3d4/linux/nvim:
    linux:
      path: ~/.config/nvim
  e5f6g7h8/.gitconfig:
    linux:
      path: ~/.gitconfig
```

不存在 `dfm.yaml` 时，内存初值为 `{"dotfiles": {}}`；已有 YAML 缺少顶层 `dotfiles` 时会补空字典。保存会创建根目录，将 YAML 写入根目录中的临时文件、`fsync` 文件后用 `os.replace` 原子替换 `dfm.yaml`，再尽力 `fsync` 目录。空文件、非映射 YAML 或错误的嵌套结构仍没有容错处理。

仓库根目录的遗留文件 **`dfm_template.yaml` 不会被读取**；源码只访问 `dfm.yaml`。

## 命令逐步实现

### `add`

1. 规范化安装路径，执行上述合法性、home 范围和保存路径冲突检查。
2. 计算保存绝对路径，创建其父目录。
3. `shutil.move(install_path, abs_save_path)` 移走原文件或目录。
4. 用 `os.symlink(abs_save_path, install_path)` 在原位置创建绝对目标链接。
5. 生成 POSIX 相对保存路径，在 `dotfiles[rel][当前系统]['path']` 写入缩写后的安装路径。
6. `main()` 将配置写入 `dfm.yaml`。

### `install`

1. 有 `<save_path>` 时，先确认其位于根目录且是现有文件或目录，转为相对保存路径，并确认当前系统在配置中有登记；否则打印 `is not kept in dotfiles`。
2. 遍历所有配置条目（或仅指定项），跳过未登记当前系统的条目；在任何覆盖提示或文件修改前，确认每个当前系统保存对象均存在且为文件或目录。
3. 用保存根目录加内部 POSIX 相对路径还原真实保存路径。
4. 调用 `_make_link(saved, install)` 创建链接；成功才打印 `Install ... -> ...`。

`_make_link` 会创建安装位置的父目录。若链接位置已存在（包含悬挂链接），它询问 `文件 ... 已存在，是否替换？(y/N)`；仅输入大小写不敏感的 `y` 才继续。`install` 会先完成整批覆盖确认，再开始替换；文件或链接用 `os.remove` 删除，目录用 `shutil.rmtree` 递归删除，然后创建符号链接。

### `share`

1. 先确认给定保存路径位于根目录且为现有文件或目录，将其转换为相对 POSIX 路径；安装目标必须位于 home 且不在根目录内。若不在配置顶层，打印 `is not kept in dotfiles` 并返回。
2. 若安装路径需要覆盖，先在事务开始前询问；拒绝则直接返回。
3. 调用同一 `_make_link` 将安装位置链接到给定保存路径；成功后才在该条目当前系统下写入新的安装路径并打印 `share ... -> ...`。
4. 若用户拒绝覆盖，不写入当前系统登记，也不重写配置。

`share` 不要求保存对象由其他平台创建；“共享”语义由已有配置条目和当前系统登记决定。但保存对象必须真实存在且为文件或目录。

### `rm`

1. 若传入路径本身是链接，用 `os.readlink` 取得其目标作为保存路径；否则直接将参数视为保存路径。
2. 计算相对 POSIX 保存路径，读取当前系统的安装路径；当前系统没有登记时静默返回。
3. 默认只删除指向该保存对象的受管链接；普通文件、目录或指向其他位置的链接会使操作失败。使用 `--force` 时，事务先快照该目标，再删除冲突对象。
4. 删除当前系统的配置项。
5. 若该保存路径仍有其他系统配置，则把保存对象复制到当前安装位置（文件 `copy`，目录 `copytree`）；否则把保存对象移动回安装位置，并删除整个保存路径配置项。
6. 若保存对象的直接父目录已空，删除该父目录；写回配置。

## 跨平台、权限与风险

* 非 Windows 使用标准 `os.path.expanduser` 展开 `~`。Windows 使用自定义逻辑：只要路径以 `~` 开头，就将第一个 `~` 替换为 `os.path.expanduser('~')`；它不只匹配 `~/` 或 `~\\`。路径缩写则只是文本前缀匹配 home 后替换为 `~`。
* Windows 上 `main()` 必须通过 `ctypes.windll.shell32.IsUserAnAdmin()` 检查；非管理员只打印提示并返回。即使通过检查，链接创建也依赖 Windows 的符号链接能力和权限。
* 保存哈希使用缩写后父目录的 UTF-8 MD5；缩写行为、home 表示及路径大小写/规范化差异会影响结果。路径范围校验使用规范化后的路径边界比较；Windows 上会忽略大小写。
* 保存路径和链接目标大多是绝对路径，迁移 home 目录或仓库后不会自动修复。

## 事务、恢复与排障

`add`、`rm`、`share` 与 `install` 的已接受批次均通过 `Transaction` 执行。变更命令在 `~/dotfiles/.dfm.lock` 上持有进程排他锁；创建快照**前**使用与恢复相同的路径验证，拒绝 root/home 外、`dfm.yaml`、锁、日志、备份目录及其子项，和已有符号链接父目录下的直接变更路径。事务状态创建也以 `lstat` 检查 root、`.dfm-backups`、日志和新 UUID 目录，拒绝通过符号链接进入元数据目录。恢复在**打开日志或清单前**再次以 `lstat` 验证 root、备份根和事务目录为真实目录，日志和 `manifest.yaml` 为普通文件；任何符号链接、非目录或非普通文件都会 fail closed。操作前将会被直接替换、移动或删除的路径，以及 `dfm.yaml`，快照到 `~/dotfiles/.dfm-backups/<UUID>/`。`rm` 还快照同一保存项的当前平台安装路径。所有快照同步后，事务在该目录写入 `manifest.yaml`，再原子写入 `~/dotfiles/.dfm-transaction.yaml`；两者必须完全匹配。提交前会同步每个直接变更的文件、目录树和父目录；YAML 和文件操作完成后，删除日志是提交点。恢复同样在删除日志前同步已恢复文件/目录树及其父目录。目录树同步以 `lstat` 跳过符号链接条目（包括悬挂链接），绝不打开或同步其目标，但会同步所在父目录。文件或目录同步失败会中止提交或恢复，保留日志和快照；目录打开或目录 `fsync` 仅明确报为不受支持（`EINVAL`、`ENOTSUP` 或 `EOPNOTSUPP`），或**仅在 Windows**目录打开报 `EACCES` 时会跳过目录同步，其他错误会传播。只有提交后 CLI 才打印成功消息。默认会删除成功事务的快照；`--backup` 保留快照和清单。

任一变更命令在读取配置前会检查日志。恢复前严格验证版本、事务 ID、根目录内的备份位置、清单匹配、配置快照、条目序号/路径及快照存在性；恢复路径只允许位于根目录或当前 home，且不能是 `dfm.yaml`、锁、日志、备份目录或其子项。完成所有日志校验后还会拒绝已有符号链接父目录，避免恢复经父目录逃逸。有效日志会先恢复路径、再以临时文件和原子替换恢复原始配置字节；快照在整个恢复成功前不会被移动或删除，因此恢复失败可重试。仅全部恢复完成并同步目标父目录后才删除日志和快照。无效或不完整日志会 fail closed：保留原文件、日志和备份，阻止自动恢复及后续变更。

运行中异常时也会立即尝试恢复；恢复失败保留日志/快照，供后续诊断。`doctor` 会报告待处理/无效事务、配置结构或路径错误、缺失保存对象，以及当前保存命名空间中未引用的保存对象和当前系统安装链接的缺失、悬挂、错误目标和“不是链接”。未引用对象扫描只进入 `get_save_path` 所产生的 32 位 MD5 顶层命名空间，不扫描 `.git`、README 或根目录其他用户文件。`doctor --repair` 只对通过上述验证的待处理事务调用恢复；若不存在其他问题，亦可用独立事务重建保存对象存在的缺失链接，或将指向其他目标的符号链接改回明确保存对象。它不会修改 YAML、补回保存对象、替换普通安装文件、猜测修复悬挂链接、清理无效日志，或清理历史 `--backup` 备份。

事务降低了中途异常造成的不一致风险，但不能消除文件系统、权限、磁盘满、进程被强制终止或手工删除日志/备份带来的风险。覆盖接受后仍会替换目标（目录会递归删除）；快照仅覆盖事务跟踪的直接路径和配置，并非整个家目录的备份。需要长期保留或跨机器恢复时，仍应自行备份整个 `~/dotfiles`。

排障建议：

1. 先执行 `dfm doctor`。若报告有效待处理事务，执行 `dfm doctor --repair`；若报告无效日志/清单，不要删除 `.dfm-transaction.yaml`、对应 `.dfm-backups/<UUID>` 或其中的 `manifest.yaml`，应先复制整个目录并人工核对。
2. 确认当前用户的 `~/dotfiles/dfm.yaml` 存在且 YAML 结构符合上述 `dotfiles → 保存路径 → 系统名 → path` 层级。
3. 使用当前 `platform.system().lower()` 对应的配置项；其他系统的登记会被 `install` 跳过。
4. 检查保存路径是否使用 `/`，以及对应的仓库对象和安装路径是否真实存在；相对链接目标会相对链接所在目录解析。
5. Windows 则以管理员身份运行。
6. 对覆盖提示默认回答 `N`，先人工检查目标类型和内容；需要恢复不一致状态时，优先使用 repair，再按配置谨慎手动恢复。
