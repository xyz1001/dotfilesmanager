# 技术实现（维护者参考）

本文依据当前的 `dotfilesmanager` 包描述行为。`dfm` 是一个以**移动原文件到集中目录，再在原安装位置创建符号链接**为工作模型的 dotfile 管理器；并非复制同步工具。

## 打包、依赖与入口

项目使用 `pyproject.toml` 打包，控制台入口为 `dotfilesmanager.cli:main`；顶层 `dfm.py` 仅保留为兼容入口。运行时依赖如下：

|依赖|用途|
|---|---|
|`docopt`|从 CLI 用法字符串解析命令行|
|`pyyaml`|以 `SafeLoader` 读取、以 `SafeDumper` 写入 YAML 配置|

`dfm setup` 是 Windows 专用的独立命令，不解析仓库、不获取锁、不打开事务或配置。其他变更命令在持锁状态下先恢复遗留事务、加载配置、完成预检并执行操作；成功提交后才输出成功消息。

## CLI 契约

```text
dfm add <install_path> [--system] [--non-interactive] [--target=SYSTEM=PATH]... [--dry-run] [--force] [--backup]
dfm rm <path> [--all] [--dry-run] [--force] [--backup]
dfm install [<save_path>] [--dry-run] [--force] [--backup]
dfm share <save_path> <install_path> [--non-interactive] [--target=SYSTEM=PATH]... [--dry-run] [--force] [--backup]
dfm doctor [--repair]
dfm setup
```

|命令|参数含义|主要结果|
|---|---|---|
|`add`|现有的安装路径；`--system` 表示保存副本与当前系统相关|移动对象入仓库，并在原位置建链接|
|`rm`|仓库保存路径，或指向仓库的符号链接路径；可加 `--all`|默认删除当前系统登记；`--all` 删除全部登记和保存对象|
|`install`|可选的仓库保存路径|按当前系统登记重建一个或全部链接|
|`share`|已有仓库保存路径与新的安装路径|把该保存对象登记给当前系统并建链接|
|`doctor`|可选 `--repair`|诊断事务、配置、保存对象和当前系统安装链接；仅恢复有效事务或安全重建明确错误的链接|

路径进入各命令前会做 `expanduser`、`normpath` 和 `abspath`。变更命令先校验 `dfm.yaml` 的顶层映射、相对保存路径、系统项及记录的 `path` 类型；所有平台记录只做这些结构校验，只有**当前平台**的安装路径才按本机 home/根目录规则校验，避免把其他系统路径当作本机错误。`add` 还要求目标是文件或目录、位于 home、未在根目录内且计算出的保存路径尚不存在。`rm` 只接受位于根目录中的路径，或解析后指向该根目录的链接，并拒绝覆盖普通文件、目录或指向其他保存对象的链接。指定的 `install`/`share` 保存路径必须位于根目录且为现有文件或目录；`share` 的安装目标必须位于 home 且不在根目录内。

`add` 和 `share` 在 TTY 默认运行 `inquirer` 目标向导；非 TTY 必须给出 `--non-interactive`。自动化调用使用重复的 `--target=SYSTEM=PATH`，平台集合和顺序固定为 `linux`、`darwin`、`windows`、`android`。目标按第一个 `=` 切分，不能是当前平台；外部路径以目标平台的 `ntpath`（Windows）或 `posixpath`（其余平台，包括 Android）作词法检查，必须为 `/` 分隔的安全 `~/...` 路径，拒绝空值、NUL、绝对/驱动器路径、`.`/`..`、home 和 `~/dotfiles`。相同现有声明幂等，不同声明始终失败，即使使用 `--force`。标准 `<md5>/<platform>/<basename>` 三段平台专用保存路径不能添加外部目标，且只可在其 owner 平台 share。向导 Checkbox 将当前平台放在首位并锁定为默认选择（仅作本地路径提示，不会成为目标）；已配置的外部平台被省略。显示名为 Linux、macOS、Windows、Android (Termux)，但 YAML 始终使用平台键。List 只转换 CONFIG 和 DATA：Windows CONFIG 为 Roaming AppData、DATA 为 Local；macOS config/data 共享 Application Support。LOG 与其他未分类来源只显示直接路径；CACHE/STATE 仅在根目录与 CONFIG/DATA 可区分时直接回退，macOS Application Support 与 Windows Local 的同根别名仍按 DATA 转换。已分类重定向来源仍显示固定类别模板和最后的直接回退，实际目标重定向由 Custom path 指定。当前来源分类可读取当前 provider 的公开 config/data root 并尊重当前 HOME/XDG/AppData；目标转换只通过固定 `platformdirs==4.3.6` 私有 append-helper 和字面 home 相对 base，绝不读取外部目标环境或 public target directory property。Android 刻意使用 Unix helper。List 只含路径候选和 Custom path，默认选中第一个候选。定制 Text 对空值或仅空白值不做路径校验，并返回该系统的路径 List；其他输入直接使用外部目标校验器。最终 Confirm 默认否。取消、EOF、缺少回答或拒绝确认均不会产生目标变更；dry-run 仅跳过最终 Confirm。回答和目标冲突会在事务开始前完成。

候选转换先使用当前平台对应的 `platformdirs` provider 分类**当前**来源：Linux/Android
使用 Unix，Darwin 使用 MacOS，Windows 的 CONFIG 使用 Roaming、其余使用 Local。公开
category root 只允许用于这一步，且会尊重当前 XDG 覆盖；只保留 config/data 的同根别名，
顺序为 config、data。随后目标转换仍只使用固定字面模板及私有 append helper：Windows
CONFIG 为 Roaming，DATA 为 Local；macOS 会先为每个识别类别提供对应 Unix 候选（CONFIG 为
`~/.config`、DATA 为 `~/.local/share`），再提供 Application Support 模板并生成去重并集，
因此 Unix 候选为默认值。其他平台在 CONFIG 模板之后额外加入 Unix `~/.config` 候选，最后才
加入直接来源回退；
LOG 和其他未分类来源只有直接回退；CACHE/STATE 只有在根目录可区分时才直接回退，
同 DATA 根的 macOS/Windows 别名仍按 DATA 处理。绝不将当前重定向/XDG/AppData 环境当作
外部机器路径，重定向目标应由 Custom path 明确输入。

路径 List 默认选中第一个生成的建议路径，末尾只有 Custom path；没有 Skip 选项。Custom path
留空或只输入空白会返回该系统的路径选择。

Android 检测优先使用 `sys.getandroidapilevel()` 返回的整数，其次要求同时存在
`ANDROID_ROOT` 和 `ANDROID_DATA`，最后识别 `platform.system() == "android"`；
`TERMUX_VERSION` 单独存在不参与检测。Android/Termux 使用唯一 `android` YAML
键，不会回退读取旧的 `linux` 映射，也不会创建 `termux` 命名空间。这是破坏性变更：
先备份并提交仓库；保留真实 Linux 共享使用的 `linux` 映射，并在 Android 上另加
`android` 映射。仅既有 Termux 专用 `linux` 映射和平台专用 `linux` 对象必须手动
重新登记或迁移。

目标等价比较也使用**目标**平台语义：Windows 忽略大小写并把历史记录中的 `\`
和新记录中的 `/` 视作同一分隔符，但保留 YAML 中已有的字符串，不为幂等比较重写
数据。Windows 来源路径在向导前会规范为 POSIX 的 `~/...`，因此
`C:\Users\Alice\.config\nvim` 也能识别 `.config` 候选。Windows 的
`~/DOTFILES/...` 同样被拒绝。

`--dry-run` 读取并校验配置，执行本命令可用的路径/保存对象预检后打印命令级预览；不获取锁、不创建根目录、不写入任何文件，也不询问覆盖。`--force` 将安装/共享覆盖确认视为接受；对 `rm` 则允许在事务快照保护下替换冲突安装目标。它不绕过保存对象、路径、日志或恢复校验。`--backup` 使成功事务保留其开始前快照及清单。

## 存储与配置生命周期

### 根目录和保存路径

根目录固定为：

```python
dotfiles_root = os.path.join(os.path.expanduser("~"), "dotfiles")  # ~/dotfiles
```

对安装路径先缩写 home（通常成为 `~`），再以其**父目录字符串**计算 MD5；最后拼接 basename。路径形式为：

```text
~/dotfiles/<md5(缩写后安装路径的dirname)>[/<platform.system().lower()>]/<basename>
```

未指定 `--system` 时，中间系统名为空，`os.path.join` 不会形成额外目录；指定后才插入如 `linux`、`darwin` 或 `windows`。因此同名但父目录不同的对象通过 MD5 目录区分；系统相关对象还通过系统名区分。

配置中的保存路径始终是相对 `~/dotfiles` 的路径，并在写入前将本机分隔符转换为内部 **POSIX `/`** 分隔符。安装时再把 `/` 转回本机 `os.sep`。

### YAML 格式

唯一读取和写入的配置文件是 `~/dotfiles/dfm.yaml`。其逻辑结构为：

```yaml
dotfiles:
  <相对保存路径（使用 /）>:
    <当前系统小写名>:
      path: <安装路径，home 可写为 ~>
```

示例（MD5 值仅为示意）：

```yaml
dotfiles:
  a1b2c3d4/linux/nvim:
    linux:
      path: ~/.config/nvim
  e5f6g7h8/.gitconfig:
    linux:
      path: ~/.gitconfig
```

不存在 `dfm.yaml` 时，内存初值为 `{"dotfiles": {}}`；已有 YAML 缺少顶层 `dotfiles` 时会补空字典。保存会创建根目录，将 YAML 写入根目录中的临时文件、`fsync` 文件后用 `os.replace` 原子替换 `dfm.yaml`，再尽力 `fsync` 目录。空文件、非映射 YAML 或错误的嵌套结构仍没有容错处理。

仓库根目录的遗留文件 **`dfm_template.yaml` 不会被读取**；源码只访问 `dfm.yaml`。

## 命令逐步实现

### `add`

1. 规范化安装路径，执行上述合法性、home 范围和保存路径冲突检查。
2. 计算保存绝对路径，创建其父目录。
3. `shutil.move(install_path, abs_save_path)` 移走原文件或目录。
4. 用 `os.symlink(abs_save_path, install_path)` 在原位置创建绝对目标链接。
5. 生成 POSIX 相对保存路径，在 `dotfiles[rel][当前系统]['path']` 写入缩写后的安装路径。
6. `main()` 将配置写入 `dfm.yaml`。

在第 1 步之后，普通 `add` 会在 TTY 为每个未配置的外部平台运行目标向导；
`--system` 跳过该步骤。非交互调用不读 stdin，而是在创建事务前解析、校验和合并
所有 `--target`。向导的最终确认、安装覆盖确认和 install 批量覆盖确认也都发生在
`Transaction.begin()` 前；事务开始后传给操作层的确认回调只会接受已经批准的计划。

### `install`

1. 有 `<save_path>` 时，先确认其位于根目录且是现有文件或目录，转为相对保存路径，并确认当前系统在配置中有登记；否则打印 `is not kept in dotfiles`。
2. 遍历所有配置条目（或仅指定项），跳过未登记当前系统的条目；在任何覆盖提示或文件修改前，确认每个当前系统保存对象均存在且为文件或目录。
3. 用保存根目录加内部 POSIX 相对路径还原真实保存路径。
4. 调用 `_make_link(saved, install)` 创建链接；成功才打印 `Install ... -> ...`。

`_make_link` 会创建安装位置的父目录。若链接位置已存在（包含悬挂链接），它询问 `文件 ... 已存在，是否替换？(y/N)`；仅输入大小写不敏感的 `y` 才继续。`install` 会先完成整批覆盖确认，再开始替换；文件或链接用 `os.remove` 删除，目录用 `shutil.rmtree` 递归删除，然后创建符号链接。

### `share`

1. 先确认给定保存路径位于根目录且为现有文件或目录，将其转换为相对 POSIX 路径；安装目标必须位于 home 且不在根目录内。若不在配置顶层，打印 `is not kept in dotfiles` 并返回。
2. 若安装路径需要覆盖，先在事务开始前询问；拒绝则直接返回。
3. 调用同一 `_make_link` 将安装位置链接到给定保存路径；成功后才在该条目当前系统下写入新的安装路径并打印 `share ... -> ...`。
4. 若用户拒绝覆盖，不写入当前系统登记，也不重写配置。

`share` 不要求保存对象由其他平台创建；“共享”语义由已有配置条目和当前系统登记决定。但保存对象必须真实存在且为文件或目录。

`share` 的本地状态矩阵为：当前登记缺失且链接正确时仅登记；登记匹配且链接正确时
no-op；登记匹配但链接缺失时重建；登记匹配但目标是普通对象或错误链接时，TTY 在
事务前询问，非交互必须使用 `--force`。已有不同当前平台路径始终失败。平台专用
保存对象跳过目标向导，拒绝显式外部目标，且只允许 owner 平台 share。CLI 在事务前
记录链接状态；状态在 `begin()` 后发生变化会抛错并回滚，绝不在事务内读取 stdin。
`install` 同样会在 `begin()` 前记录每个已批准安装目标的链接状态，并在实际替换前
重新核对；状态变化会使事务回滚，不会在事务内重新确认。

### `rm`

1. 若传入路径本身是链接，用 `os.readlink` 取得其目标作为保存路径；否则直接将参数视为保存路径。
2. 计算相对 POSIX 保存路径，读取当前系统的安装路径；默认情况下当前系统没有登记时静默返回。`--all` 则删除保存对象和整个配置项，且不读取或访问任何外部平台安装路径。
3. 默认只删除指向该保存对象的受管链接；普通文件、目录或指向其他位置的链接会使操作失败。使用 `--force` 时，事务先快照该目标，再删除冲突对象。
4. 删除当前系统的配置项；`--all` 删除整个配置项。
5. 默认时若该保存路径仍有其他系统配置，则把保存对象复制到当前安装位置（文件 `copy`，目录 `copytree`）；否则把保存对象移动回安装位置，并删除整个保存路径配置项。`--all` 在当前系统已登记时始终把保存对象移动回该本地安装位置，然后删除整个配置项。
6. 写回配置；保存对象的空父目录保留，以确保事务只跟踪直接变更路径。

## 跨平台、权限与风险

* 非 Windows 使用标准 `os.path.expanduser` 展开 `~`。Windows 使用自定义逻辑：只要路径以 `~` 开头，就将第一个 `~` 替换为 `os.path.expanduser('~')`；它不只匹配 `~/` 或 `~\\`。路径缩写则只是文本前缀匹配 home 后替换为 `~`。
* Windows 普通命令不要求管理员。每次用户可见的链接创建保留 `target_is_directory`，且仅将 Windows `ERROR_PRIVILEGE_NOT_HELD`（WinError 1314）分类为可通过 `dfm setup` 处理的失败；其他原生错误保持不变。`setup` 先在临时目录实际测试文件和目录链接。仅当该测试得到 1314 时，才在 64 位注册表视图设置 `HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\AppModelUnlock\AllowDevelopmentWithoutDevLicense` DWORD 为 `1`，不设置 `AllowAllTrustedApps`。非提升进程只通过 UAC 提升绝对系统目录的 `reg.exe` 子进程及固定参数，等待其退出、读取设置并再次探测；取消、子进程失败、读取或复测失败均不报告成功。
* 保存哈希使用缩写后父目录的 UTF-8 MD5；缩写行为、home 表示及路径大小写/规范化差异会影响结果。路径范围校验使用规范化后的路径边界比较；Windows 上会忽略大小写。
* 保存路径和链接目标大多是绝对路径，迁移 home 目录或仓库后不会自动修复。

## 事务、恢复与排障

`add`、`rm`、`share` 与 `install` 的已接受批次均通过 `Transaction` 执行。变更命令在 `~/dotfiles/.dfm.lock` 上持有进程排他锁；创建快照**前**使用与恢复相同的路径验证，拒绝 root/home 外、`dfm.yaml`、锁、日志、备份目录及其子项，和已有符号链接父目录下的直接变更路径。事务状态创建也以 `lstat` 检查 root、`.dfm-backups`、日志和新 UUID 目录，拒绝通过符号链接进入元数据目录。恢复在**打开日志或清单前**再次以 `lstat` 验证 root、备份根和事务目录为真实目录，日志和 `manifest.yaml` 为普通文件；任何符号链接、非目录或非普通文件都会 fail closed。操作前将会被直接替换、移动或删除的路径，以及 `dfm.yaml`，快照到 `~/dotfiles/.dfm-backups/<UUID>/`。`rm` 只快照保存对象及同一保存项的当前平台安装路径；即使使用 `--all` 也绝不快照或访问外部平台安装路径。所有快照同步后，事务在该目录写入 `manifest.yaml`，再原子写入 `~/dotfiles/.dfm-transaction.yaml`；两者必须完全匹配。提交前会同步每个直接变更的文件、目录树和父目录；YAML 和文件操作完成后，删除日志是提交点。恢复同样在删除日志前同步已恢复文件/目录树及其父目录。目录树同步以 `lstat` 跳过符号链接条目（包括悬挂链接），绝不打开或同步其目标，但会同步所在父目录。文件或目录同步失败会中止提交或恢复，保留日志和快照；目录打开或目录 `fsync` 仅明确报为不受支持（`EINVAL`、`ENOTSUP` 或 `EOPNOTSUPP`），或**仅在 Windows**目录打开报 `EACCES` 时会跳过目录同步，其他错误会传播。只有提交后 CLI 才打印成功消息。默认会删除成功事务的快照；`--backup` 保留快照和清单。

任一变更命令在读取配置前会检查日志。恢复前严格验证版本、事务 ID、根目录内的备份位置、清单匹配、配置快照、条目序号/路径及快照存在性；恢复路径只允许位于根目录或当前 home，且不能是 `dfm.yaml`、锁、日志、备份目录或其子项。完成所有日志校验后还会拒绝已有符号链接父目录，避免恢复经父目录逃逸。有效日志会先恢复路径、再以临时文件和原子替换恢复原始配置字节；快照在整个恢复成功前不会被移动或删除，因此恢复失败可重试。仅全部恢复完成并同步目标父目录后才删除日志和快照。无效或不完整日志会 fail closed：保留原文件、日志和备份，阻止自动恢复及后续变更。

运行中异常时也会立即尝试恢复；恢复失败保留日志/快照，供后续诊断。`doctor` 会报告待处理/无效事务、配置结构或路径错误、缺失保存对象，以及当前保存命名空间中未引用的保存对象和当前系统安装链接的缺失、悬挂、错误目标和“不是链接”。未引用对象扫描只进入 `get_save_path` 所产生的 32 位 MD5 顶层命名空间，不扫描 `.git`、README 或根目录其他用户文件。`doctor --repair` 只对通过上述验证的待处理事务调用恢复；若不存在其他问题，亦可用独立事务重建保存对象存在的缺失链接，或将指向其他目标的符号链接改回明确保存对象。它不会修改 YAML、补回保存对象、替换普通安装文件、猜测修复悬挂链接、清理无效日志，或清理历史 `--backup` 备份。

事务降低了中途异常造成的不一致风险，但不能消除文件系统、权限、磁盘满、进程被强制终止或手工删除日志/备份带来的风险。覆盖接受后仍会替换目标（目录会递归删除）；快照仅覆盖事务跟踪的直接路径和配置，并非整个家目录的备份。需要长期保留或跨机器恢复时，仍应自行备份整个 `~/dotfiles`。

排障建议：

1. 先执行 `dfm doctor`。若报告有效待处理事务，执行 `dfm doctor --repair`；若报告无效日志/清单，不要删除 `.dfm-transaction.yaml`、对应 `.dfm-backups/<UUID>` 或其中的 `manifest.yaml`，应先复制整个目录并人工核对。
2. 确认当前用户的 `~/dotfiles/dfm.yaml` 存在且 YAML 结构符合上述 `dotfiles → 保存路径 → 系统名 → path` 层级。
3. 使用当前 `platform.system().lower()` 对应的配置项；其他系统的登记会被 `install` 跳过。
4. 检查保存路径是否使用 `/`，以及对应的仓库对象和安装路径是否真实存在；相对链接目标会相对链接所在目录解析。
5. Windows 若链接创建报 1314，运行 `dfm setup` 后重试；其他 Windows 错误按原始错误排查。
6. 对覆盖提示默认回答 `N`，先人工检查目标类型和内容；需要恢复不一致状态时，优先使用 repair，再按配置谨慎手动恢复。
