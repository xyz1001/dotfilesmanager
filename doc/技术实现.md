# 技术实现（维护者参考）

本文只依据当前的 `dfm.py` 与 `setup.py` 描述行为。`dfm` 是一个以**移动原文件到集中目录，再在原安装位置创建符号链接**为工作模型的 dotfile 管理器；并非复制同步工具。

## 打包、依赖与入口

`setup.py` 将顶层模块 `dfm.py` 作为 `py_modules=['dfm']` 打包，包名为 `dotfilesmanager`，版本 `1.1.2`。运行时依赖如下：

|依赖|用途|
|---|---|
|`docopt`|从 `dfm.py` 顶部文档字符串解析命令行|
|`pyyaml`|以 `SafeLoader` 读取、以 `SafeDumper` 写入 YAML 配置|

控制台入口是：

```ini
dfm = dfm:main
```

`main()` 在 Windows 先检查管理员权限，随后解析 CLI、加载配置、分派一个操作函数并无条件写回配置。

## CLI 契约

```text
dfm add <install_path> [--system]
dfm rm <path>
dfm install [<save_path>]
dfm share <save_path> <install_path>
```

|命令|参数含义|主要结果|
|---|---|---|
|`add`|现有的安装路径；`--system` 表示保存副本与当前系统相关|移动对象入仓库，并在原位置建链接|
|`rm`|仓库保存路径，或指向仓库的符号链接路径|删除当前系统的登记；恢复文件或保留共享副本|
|`install`|可选的仓库保存路径|按当前系统登记重建一个或全部链接|
|`share`|已有仓库保存路径与新的安装路径|把该保存对象登记给当前系统并建链接|

路径进入各命令前会做 `__expanduser`、`normpath` 和 `abspath`。`add` 要求目标是文件或目录、位于 home 目录下、不能位于 dotfiles 根目录中，并且计算出的保存路径尚不存在。`rm` 只接受位于 dotfiles 根目录中的路径，或解析后指向该根目录的链接。`install` 与 `share` 没有同等完整的输入存在性校验。

## 存储与配置生命周期

### 根目录和保存路径

根目录固定为：

```python
dotfiles_root = os.path.join(os.path.expanduser("~"), "dotfiles")  # ~/dotfiles
```

对安装路径先缩写 home（通常成为 `~`），再以其**父目录字符串**计算 MD5；最后拼接 basename。路径形式为：

```text
~/dotfiles/<md5(缩写后安装路径的dirname)>[/<platform.system().lower()>]/<basename>
```

未指定 `--system` 时，中间系统名为空，`os.path.join` 不会形成额外目录；指定后才插入如 `linux`、`darwin` 或 `windows`。因此同名但父目录不同的对象通过 MD5 目录区分；系统相关对象还通过系统名区分。

配置中的保存路径始终是相对 `~/dotfiles` 的路径，并在写入前将本机分隔符转换为内部 **POSIX `/`** 分隔符。安装时再把 `/` 转回本机 `os.sep`。

### YAML 格式

唯一读取和写入的配置文件是 `~/dotfiles/dfm.yaml`。其逻辑结构为：

```yaml
dotfiles:
  <相对保存路径（使用 /）>:
    <当前系统小写名>:
      path: <安装路径，home 可写为 ~>
```

示例（MD5 值仅为示意）：

```yaml
dotfiles:
  a1b2c3d4/linux/nvim:
    linux:
      path: ~/.config/nvim
  e5f6g7h8/.gitconfig:
    linux:
      path: ~/.gitconfig
```

不存在 `dfm.yaml` 时，内存初值为 `{"dotfiles": {}}`；已有 YAML 缺少顶层 `dotfiles` 时会补空字典。每次成功走完分派后都会调用保存，使用普通文本写入（`newline='\n'`），不是原子替换。仓库根目录不会由加载/保存自动创建：例如首次直接执行 `install`、`rm` 或 `share`，若 `~/dotfiles` 尚不存在，最终写配置可能失败。空文件、非映射 YAML 或错误的嵌套结构也没有容错处理。

仓库根目录的遗留文件 **`dfm_template.yaml` 不会被读取**；源码只访问 `dfm.yaml`。

## 命令逐步实现

### `add`

1. 规范化安装路径，执行上述合法性、home 范围和保存路径冲突检查。
2. 计算保存绝对路径，创建其父目录。
3. `shutil.move(install_path, abs_save_path)` 移走原文件或目录。
4. 用 `os.symlink(abs_save_path, install_path)` 在原位置创建绝对目标链接。
5. 生成 POSIX 相对保存路径，在 `dotfiles[rel][当前系统]['path']` 写入缩写后的安装路径。
6. `main()` 将配置写入 `dfm.yaml`。

### `install`

1. 有 `<save_path>` 时，先转绝对路径、转为相对保存路径，并确认当前系统在配置中有登记；否则打印 `is not kept in dotfiles` 并仍保存配置。
2. 遍历所有配置条目（或仅指定项），跳过未登记当前系统的条目。
3. 用保存根目录加内部 POSIX 相对路径还原真实保存路径。
4. 调用 `__mklink(saved, install)` 创建链接；成功才打印 `Install ... -> ...`。

`__mklink` 会创建安装位置的父目录。若链接位置已存在（包含悬挂链接），它询问 `文件 ... 已存在，是否替换？(y/N)`；仅输入大小写不敏感的 `y` 才继续。文件或链接用 `os.remove` 删除，目录用 `shutil.rmtree` 递归删除，然后创建符号链接。

### `share`

1. 将给定保存路径转换为相对 POSIX 路径；若不在配置顶层，打印 `is not kept in dotfiles` 并返回。
2. 立即在该条目当前系统下写入新的安装路径。
3. 调用同一 `__mklink` 将安装位置链接到给定保存路径；成功打印 `share ... -> ...`。
4. 无论用户在覆盖提示中拒绝与否，步骤 2 的登记都会在 `main()` 中写入配置。

`share` 不验证保存对象实际存在，也不要求它是其他平台创建的；“共享”语义由已有配置条目和当前系统登记决定。

### `rm`

1. 若传入路径本身是链接，用 `os.readlink` 取得其目标作为保存路径；否则直接将参数视为保存路径。
2. 计算相对 POSIX 保存路径，读取当前系统的安装路径；当前系统没有登记时静默返回。
3. 若安装位置仍是符号链接，先 `os.unlink` 删除它；普通文件/目录不会在此删除。
4. 删除当前系统的配置项。
5. 若该保存路径仍有其他系统配置，则把保存对象复制到当前安装位置（文件 `copy`，目录 `copytree`）；否则把保存对象移动回安装位置，并删除整个保存路径配置项。
6. 若保存对象的直接父目录已空，删除该父目录；写回配置。

## 跨平台、权限与风险

* 非 Windows 使用标准 `os.path.expanduser` 展开 `~`。Windows 使用自定义逻辑：只要路径以 `~` 开头，就将第一个 `~` 替换为 `os.path.expanduser('~')`；它不只匹配 `~/` 或 `~\\`。路径缩写则只是文本前缀匹配 home 后替换为 `~`。
* Windows 上 `main()` 必须通过 `ctypes.windll.shell32.IsUserAnAdmin()` 检查；非管理员只打印提示并返回。即使通过检查，链接创建也依赖 Windows 的符号链接能力和权限。
* 保存哈希使用缩写后父目录的 UTF-8 MD5；缩写行为、home 表示及路径大小写/规范化差异会影响结果。路径校验使用字符串 `startswith`，不是路径边界感知比较。
* 保存路径和链接目标大多是绝对路径，迁移 home 目录或仓库后不会自动修复。

## 已知限制、风险与排障

该实现没有事务、备份或 dry-run。`add` 在移动成功后若建链接或写 YAML 失败，可能留下已移动但未登记的对象；`rm`、复制和移动也可能在中途失败。不要把交互式覆盖当作安全确认：`install`/`share` 接受 `y` 后会永久删除已有文件，或对目录执行递归 `rmtree`，不会备份。操作前应自行备份目标和 `~/dotfiles/dfm.yaml`。

排障建议：

1. 确认当前用户的 `~/dotfiles/dfm.yaml` 存在且 YAML 结构符合上述 `dotfiles → 保存路径 → 系统名 → path` 层级。
2. 使用当前 `platform.system().lower()` 对应的配置项；其他系统的登记会被 `install` 跳过。
3. 检查保存路径是否使用 `/`，以及对应的仓库对象和安装路径是否真实存在；`readlink` 得到相对链接目标时，当前实现不会相对链接所在目录重新解析。
4. 遇到首次运行写配置失败，手动创建 `~/dotfiles` 后重试；Windows 则以管理员身份运行。
5. 对覆盖提示默认回答 `N`，先人工检查目标类型和内容；需要恢复不一致状态时，按配置定位保存对象，再谨慎手动移动/重建链接。
